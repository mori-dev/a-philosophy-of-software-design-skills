# APoSD PR コメント テンプレート — 日本語

このファイルは、PR レビューで使用する日本語のコメント テンプレートを提供します。

各テンプレートは、要指摘の種類に応じて使い分けてください。日本語の技術文書ルール（writing-style.md など）に従い、即物的・正確な表現を心がけています。

## 浅いモジュール (Shallow Module)

**適用場面**: インタフェースが複雑な割に実装が単純、もしくは呼び出し側がモジュール内ロジックを知りすぎている

### ブロッキング版（このPRで直す）

```
## 浅いモジュール — `file.py:line_range`

このレポジトリクラスは、SQL をそのまま露出しているため、インタフェースが複雑な割に実装が単純です。

**状況**:
- メソッドが複数あるが、それぞれ 1 行のクエリを返すだけ
- 呼び出し側は直接 SQL を意識する必要がある

**影響**:
- ユーザー テーブルのスキーマ変更が、クエリのすべてのメソッドに波及する
- テストが DB に依存する

**提案**:
モジュールを深くして、呼び出し側から SQL を隠してください。

例：
```python
# 変更前
def get_active_users(self):
    return db.query("SELECT * FROM users WHERE status = 'active'")
def get_by_email(self, email):
    return db.query("SELECT * FROM users WHERE email = ?", email)

# 変更後
def find_active(self):
    # 内部で SQL、キャッシュ、検証を隠す
    return self._query_users(status='active')
def find_by_email(self, email):
    # インタフェースは何をするかだけ明確
    return self._query_users(email=email)
```

テスト可能性も改善されます（モック化しやすくなる）。
```

### 非ブロッキング版（後続でよい）

```
## 浅いモジュール — 検討対象 `file.py:line_range`

このサービス層は、複数のメソッドでリポジトリにそのまま転送しているだけです。

**パターン**:
```python
def get_user(self, id):
    return self.repository.get_user(id)  # 転送するだけ
```

**後続で検討すべき**:
- このサービス層が本当に必要か（リポジトリを直接呼べば十分でないか）
- もし必要なら、キャッシュ、ログ、バリデーション など、実質的な責務を持たせる

今回のPRでは修正不要ですが、リファクタ計画で優先度を上げる候補です。
```

## 情報漏洩 (Information Leakage)

**適用場面**: 呼び出し側が内部構造を知りすぎている（型、DB スキーマ、ストレージ形式など）

### ブロッキング版

```
## 情報漏洩 — `file.py:line_range`

呼び出し側が内部の ユーザー ステータス enum を知りすぎています。

**状況**:
```python
from models import User, UserStatus  # 内部型を露出
user = repo.get(id)
if user.status == UserStatus.PENDING:  # ステータスの内部表現に依存
    ...
```

**影響**:
- ユーザー ステータスが変わると、呼び出し側のコードも変わる必要がある
- DB スキーマ変更がレイアでに波及する

**提案**:
リポジトリに確認メソッドを追加して、呼び出し側から enum を隠してください。

```python
# 変更後
user = repo.find_verified(id)
if user:  # repo が検証ロジックを持つ
    # 呼び出し側は status enum を知らない
```

この方法で、内部のステータス表現が変わっても、呼び出し側は変わりません。
```

### 非ブロッキング版

```
## 情報漏洩 — 検討対象 `file.py:line_range`

API レスポンスが DB スキーマをそのまま反映しています。

**パターン**:
```python
{
  "user_id": 123,
  "user_name": "Alice",
  "created_at": "2024-01-01T00:00:00Z"
}
```

このフォーマットが DB テーブル設計に即座に結びついています。

**後続で検討すべき**:
- API コントラクトを DB スキーマから分離する
- 必要なフィールドだけ返す API 設計へ

今のPRでは不要ですが、API 設計のリファクタとして検討ください。
```

## 変更波及 (Change Amplification)

**適用場面**: 1 つの論理的変更が複数のファイルに波及する

### ブロッキング版

```
## 変更波及 — `file_a.py`, `file_b.py`, `file_c.py`

ユーザーメール フィールドの追加が 5 つのファイルに波及しています。

**状況**:
- `models.py`: User モデルにメールフィールド
- `repository.py`: SQL クエリを修正
- `service.py`: バリデーションロジック追加
- `api.py`: レスポンスに email 含める
- `test.py`: テストを複数更新

**影響**:
この設計では、ユーザー型に 1 フィールド追加するだけで、レイアが変わってしまいます。

**提案**:
レポジトリが DB スキーマを完全に隠すように設計を改めてください。API 層は User 型ではなく、レスポンス DTO を返すようにします。

```python
# 変更後
# repository は User 型を返すが、API スキーマからは独立
# API layer
user = repository.find(id)
return UserResponse(
    id=user.id,
    email=user.email,
    # DB スキーマ変更はここだけ
)
```

このようにレイアを分けることで、DB 変更が API に波及しません。
```

### 非ブロッキング版

```
## 変更波及 — 検討対象 `multiple files`

ユーザーステータス をミドルウェアとコントローラーにコピーしています。

**パターン**:
同じロジックが複数の場所にあると、変更時に複数場所を修正する必要が生じます。

**後続で検討すべき**:
- 共通ロジックを 1 か所に集約する（中核レポジトリまたはサービス）

この PR では必要ありません。次のリファクタ PR で対応してください。
```

## 不明確な命名 (Vague Naming)

**適用場面**: 関数・クラス名が何をするかを明確に表していない

### ブロッキング版

```
## 命名が不明確 — `file.py:line_range`

関数名 `process_data()` では何をしているか不明です。

**現在**:
```python
def process_data(items):
    return [x * 2 for x in items if x > 10]
```

何を処理しているのか、コードを読まないと分かりません。

**提案**:
実際の処理内容を関数名に反映させてください。

```python
def filter_and_double_large_items(items):
    return [x * 2 for x in items if x > 10]
```

または：
```python
def double_items_over_threshold(items, threshold=10):
    return [x * 2 for x in items if x > threshold]
```

こうすることで、呼び出し側が関数の意図を一瞬で理解できます。
```

### 非ブロッキング版

```
## 命名が不明確 — 検討対象 `file.py:line_range`

メソッド `convert()` の名前が曖昧です。何に変換しているのか不明です。

**提案**:
```python
# 改善案
def to_api_response(user):
    # 内部 User型 → API レスポンス形式
    ...
```

このような詳細な命名により、コードの意図が明確になります。

今の PR では必須ではありませんが、次の機会にご検討ください。
```

## 特殊ケース の混在 (Special-Case Mixture)

**適用場面**: 呼び出し側や文脈に応じた分岐がコード内に散乱している

### ブロッキング版

```
## 特殊ケースの混在 — `file.py:line_range`

関数 `save_user()` が、呼び出し側に応じた複数の分岐を持っています。

**状況**:
```python
def save_user(user, is_admin=False, skip_validation=False):
    if not skip_validation:
        validate(user)
    if is_admin:
        # 管理者用ロジック
        ...
    else:
        # 通常ロジック
        ...
```

呼び出し側が `is_admin` フラグを渡して、動作を切り替えています。

**影響**:
- テストケースが増える（フラグ組み合わせの爆発）
- 新しい呼び出し側が新しい特殊ケースを追加する傾向
- ロジックが混乱する

**提案**:
呼び出し側ごとに明確な、別々のインタフェースを提供してください。

```python
def save_user(user):
    # 標準的な、検証が入る経路
    validate(user)
    ...

def save_user_admin_override(user):
    # 管理者用、明示的で別の関数
    # ここだけで管理者ロジック
    ...
```

呼び出し側が使うインタフェースで意図が明確になります。
```

### 非ブロッキング版

```
## 特殊ケースの混在 — 検討対象 `file.py:line_range`

設定に応じた分岐が複数あります。

**パターン**:
```python
if config.enable_caching:
    # キャッシュ有効時
    ...
elif config.enable_fallback:
    # フォールバック有効時
    ...
```

**後続で検討すべき**:
- 完全に分離した実装クラスに分ける
- またはファクトリで設定に応じたインスタンスを作成する

このPR では必要ありません。次のリファクタでご検討ください。
```

## エラーハンドリング複雑性 (Unnecessary Error-Path Complexity)

**適用場面**: エラー型が多すぎたり、try-catch が散乱していたり、エラーが握りつぶされている

### ブロッキング版

```
## エラーハンドリングの複雑性 — `file.py:line_range`

呼び出し側が複数の例外型をキャッチする必要があります。

**状況**:
```python
try:
    user = repository.get_user(id)
except UserNotFoundError:
    user = None
except DatabaseError:
    raise
```

呼び出し側が `UserNotFoundError` を知る必要があります。

**提案**:
リポジトリから例外を消す設計に変えてください。None を返す、または Optional を使う。

```python
# 変更後
user = repository.find_user(id)  # 存在しなければ None
if not user:
    # 呼び出し側で適切に処理
    ...
```

これで呼び出し側は例外型を知らなくてすみます。
```

### 非ブロッキング版

```
## エラーハンドリング複雑性 — 検討対象 `file.py:line_range`

エラーハンドリングが複数のレイアに分散しています。

**パターン**:
- `service.py` で try-catch
- `controller.py` でも try-catch
- 同じエラーを複数の場所で処理

**後続で検討すべき**:
- エラー処理を 1 か所に集約する
- 呼び出し側が知らなくてもいい例外型は消す

今の PR では必須ではありませんが、今後のリファクタで優先度を上げてください。
```

## 過度な設定 (Excessive Configuration)

**適用場面**: パラメータ・フラグ・オプションが多すぎる

### ブロッキング版

```
## 過度な設定 — `file.py:line_range`

キャッシュ の初期化パラメータが多すぎます。

**状況**:
```python
cache = Cache(
    ttl=300,
    eviction_policy='lru',
    distributed=True,
    compression=True,
    encryption=True,
    metrics_enabled=True
)
```

呼び出し側が 6 つの決定を下す必要があります。

**提案**:
内部で判断を固める、または 2 つの明確なクラスに分ける。

```python
# 変更後：シンプルな設定
cache = LocalCache(ttl=300)

# または、明示的で複雑なバージョン
cache = DistributedSecureCache(ttl=300)  # 暗号化などはデフォルト
```

呼び出し側は選択肢を減らせます。
```

### 非ブロッキング版

```
## 過度な設定 — 検討対象 `file.py:line_range`

新しい boolean フラグを追加しています。

**提案**:
無効な設定の組み合わせがないか確認してください。フラグ数が増えると、呼び出し側の認知負荷が上がります。

今の PR では不要ですが、設定の合理化をご検討ください。
```

## 使用上の注意

1. **テンプレートは調整する**: 各コメントは状況に応じてカスタマイズしてください。形式を無理に守る必要はありません。
2. **1PR で多数の指摘をしない**: 指摘が多すぎると、PR 作成者が圧倒されます。最大 3-4 件に絞る。
3. **ブロッキング/非ブロッキングを区別**: 何が今 PR で必須か、何が後でいいか明確にします。
4. **理由を付ける**: "こうしてください" だけでなく、"なぜそれが重要か" を説明します。
5. **批判ではなく、改善提案**: 「悪い」ではなく「こうすると保守性が上がります」と書きます。
